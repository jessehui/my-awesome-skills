# verify-claims Skill 设计文档

## 概述

创建一个通用 skill，在 Claude 输出包含关键断言时自动验证其正确性，或由用户手动触发验证。通过内联标注呈现验证结果，确保结论建立在可靠的证据之上。

- **名称**: `verify-claims`
- **位置**: `~/.claude/skills/verify-claims/SKILL.md`
- **适用范围**: 通用，所有项目

## 触发条件

### 自动触发

当 Claude 的输出包含事实性声明、因果推断或技术建议，且这些断言是后续结论的基础时，自动进入验证流程。

### 用户触发

用户说"验证一下"、"check this"、"verify" 等指令时，Claude 对已有输出进行验证，并允许用户指定需要额外验证的具体断言。

### 不触发

纯代码生成、简单问答（"什么是 TCP"）、闲聊、用户明确说跳过验证时。

## 关键断言识别与分类

Claude 在输出过程中识别以下三类需要验证的断言：

| 类型 | 示例 | 判定为"关键"的标准 |
|------|------|-------------------|
| **事实性声明** | "Megatron-LM v0.15.3 不再依赖 Apex" | 具体版本号、日期、数值、API 行为、工具特性 |
| **因果推断** | "使用 HugePages 可以降低 TLB miss 从而提升训练吞吐" | 后续结论或建议依赖此推理链 |
| **技术建议** | "建议用 NCCL_SOCKET_IFNAME=eth0 来指定通信网卡" | 用户可能直接执行，错误会造成实际影响 |

### 筛选规则（自动触发时）

- 后续结论依赖该断言 → 必须验证
- 断言出错会导致用户执行错误操作 → 必须验证
- 属于 Claude 已知容易产生幻觉的领域（版本号、具体参数值、兼容性矩阵）→ 必须验证
- 广泛已知的常识（"Linux 是开源的"）→ 跳过

### 用户触发时

除以上自动筛选外，用户可指定任意断言要求验证。

## 验证方法与优先级

### 步骤一：按断言类型选择验证路径

| 断言类型 | 首选来源 | 次选来源 |
|----------|----------|----------|
| 项目相关事实（"我们的 driver 是 580.159.04"） | 本地文件、代码、git 历史、CLAUDE.md | — 无需外部搜索 |
| 技术版本/API/兼容性（"PyTorch 2.6 支持 CUDA 12.4"） | 本地文档（如有） | anysearch 搜索官方文档 |
| 因果推断（"isolcpus 减少调度抖动"） | 本地已有的实验数据或文档 | anysearch 搜索技术文章、论文、官方说明，寻找多源交叉验证 |
| 技术建议（"建议设置 NCCL_IB_DISABLE=1"） | 本地配置和历史 | anysearch 搜索官方文档确认参数含义和适用条件 |

### 步骤二：执行验证

- 本地验证：读文件、grep 代码、查 git log
- 外部验证：通过 anysearch 搜索，优先官方文档和一手来源
- 因果推断需要至少找到一个独立来源支撑逻辑链条

### 步骤三：判定结果

- **已验证**：找到可靠来源明确支持
- **部分验证**：找到相关信息但不完全匹配，或来源可靠性一般
- **未验证**：未找到支撑，或找到矛盾信息

## 标注格式

在正文中对关键断言进行内联标注，使用彩色 emoji 区分：

```
[✅已验证: 来源说明]
[⚠️部分验证: 说明]
[❗未验证: 说明]
```

### 示例

```
Megatron-LM v0.15.3 不再依赖 Apex [✅已验证: pyproject.toml 中无 apex 依赖]

使用 HugePages 可以降低 TLB miss 从而提升训练吞吐 [⚠️部分验证: TLB miss 降低有据，训练吞吐提升幅度因场景而异]

建议设置 NCCL_ALGO=Ring 以获得最佳性能 [❗未验证: 官方文档未推荐固定算法，NCCL 自动选择通常更优]
```

### 标注原则

- 来源说明尽量简短，一句话点明出处或原因
- 标注紧跟在断言之后，不另起段落
- 如果验证发现断言有误，直接修正断言内容并标注修正原因，而非保留错误断言

## 自动触发 vs 用户触发的流程差异

### 自动触发流程

1. Claude 生成回答时，识别关键断言（按筛选规则）
2. 对关键断言执行验证（按验证方法）
3. 在输出中内联标注验证结果
4. 非关键断言正常输出，不标注

### 用户触发流程

1. 用户对已有输出说"验证一下"或指定具体断言
2. Claude 先按自动模式的筛选规则识别所有关键断言
3. 再加入用户指定的额外断言（即使不满足"关键"标准）
4. 对全部目标断言执行验证
5. 输出完整的带标注版本

### 关键区别

- 自动模式：只验证影响结论的关键断言，保持输出效率
- 用户模式：关键断言 + 用户指定的任意断言，覆盖更全
